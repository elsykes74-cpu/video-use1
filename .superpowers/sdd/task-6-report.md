  - `_collect_scene_images` now takes `run_id`/`root` and, on a weak match, calls `_resolve_weak_match_with_approval` instead of raising immediately.

- Modified `quickkick_bot/morning_runner.py`: the failure-notification branch now distinguishes an approval cancellation (🛑 "Run cancelled (weak-match approval)") from a generic failure (❌), since both currently surface as the same `except Exception` in `main()`.

## Operational requirement (not a code change, please read)

The approval handshake is **inter-process**: whatever calls `_run_pipeline_sync` (e.g. `morning_runner.py`, a short-lived script run once a day by Task Scheduler) blocks in `wait_for_approval`, polling `_runs/<run_id>/approval_state.json`. The `/approve_run` / `/reject_run` Telegram commands are only received by whichever process is running `quickkick_bot.pipeline`'s own bot polling loop (`_async_main()` / `main()`). **That bot process must already be running and online in Telegram at the moment a weak-match pause happens**, or there is nothing to receive the reply with. If it's not running, the run will simply time out after `approval_timeout_seconds` with no way to approve it. This is a deployment requirement, not something fixable in this module alone — flagging it because it wasn't called out elsewhere in the design doc.

## Concerns

- `request_approval`'s Telegram alert sends the *materialized* match-plan images (i.e., whatever was selected, weak or not) as the preview — this matches "generate a full contact-sheet preview of the selected image set" in the design doc.
- The design doc's approval channel is specifically "`@TheKingLives_bot`" and the original task brief pointed at a `bot/bot.py` package — neither exists in this worktree (only in the separate `C:\Users\erick\video-use1\bot` checkout, which is a different git history entirely). The approval gate is implemented against `quickkick_bot/pipeline.py`'s own embedded bot instead. Whichever `BOT_TOKEN` is configured in `.env` is the bot that will actually receive `/approve_run` — if the user wants this to literally be `@TheKingLives_bot`, that bot's token needs to be the one in `quickkick_bot/.env`'s `BOT_TOKEN`, or the approval commands need to be ported into the separate `bot/bot.py` package instead (cross-repo work, out of scope for this worktree).
- `_materialize_match_plan`'s last-good-image reuse fallback (for scenes with truly no candidate) is a judgment call to keep an approved run renderable; if every scene is missing a candidate this still raises since there's no `last_good` to reuse.
